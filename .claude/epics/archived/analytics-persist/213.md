## Technical Details

### `data/repository.py` Changes

Add 6 methods under `# --- Analytics: Queries ---`:

1. **`get_win_rate_by_direction() -> list[WinRateResult]`**
```sql
SELECT rc.direction,
       COUNT(*) as total_contracts,
       SUM(CASE WHEN co.is_winner = 1 THEN 1 ELSE 0 END) as winners,
       SUM(CASE WHEN co.is_winner = 0 THEN 1 ELSE 0 END) as losers,
       AVG(co.is_winner) as win_rate
FROM recommended_contracts rc
JOIN contract_outcomes co ON co.recommended_contract_id = rc.id
GROUP BY rc.direction
```

2. **`get_score_calibration(bucket_size: float = 10.0) -> list[ScoreCalibrationBucket]`**
```sql
SELECT
    CAST(rc.composite_score / ? AS INTEGER) * ? as score_min,
    CAST(rc.composite_score / ? AS INTEGER) * ? + ? as score_max,
    COUNT(*) as contract_count,
    AVG(co.contract_return_pct) as avg_return_pct,
    AVG(co.is_winner) as win_rate
FROM recommended_contracts rc
JOIN contract_outcomes co ON co.recommended_contract_id = rc.id
WHERE co.contract_return_pct IS NOT NULL
GROUP BY CAST(rc.composite_score / ? AS INTEGER)
ORDER BY score_min
```

3. **`get_indicator_attribution(indicator: str, holding_days: int = 5) -> list[IndicatorAttributionResult]`**
   - This query is more complex — it needs to JOIN with `normalization_metadata` or `ticker_scores`
   - Query pattern: for each scan, get the indicator's normalized value from `ticker_scores.signals_json`, correlate with `contract_outcomes.contract_return_pct`
   - May need to parse JSON in SQLite (`json_extract()`) or do the computation in Python
   - **Recommended approach:** Do the JOIN in SQL to get (indicator_value, return_pct) pairs, compute correlation in Python via numpy

4. **`get_optimal_holding_period(direction: SignalDirection | None = None) -> list[HoldingPeriodResult]`**
```sql
SELECT co.holding_days,
       rc.direction,
       AVG(co.contract_return_pct) as avg_return_pct,
       -- median requires window function or Python
       AVG(co.is_winner) as win_rate,
       COUNT(*) as sample_size
FROM recommended_contracts rc
JOIN contract_outcomes co ON co.recommended_contract_id = rc.id
WHERE co.contract_return_pct IS NOT NULL
  AND (? IS NULL OR rc.direction = ?)
GROUP BY co.holding_days, rc.direction
ORDER BY co.holding_days
```

5. **`get_delta_performance(bucket_size: float = 0.1, holding_days: int = 5) -> list[DeltaPerformanceResult]`**
```sql
SELECT
    CAST(rc.delta / ? AS INTEGER) * ? as delta_min,
    CAST(rc.delta / ? AS INTEGER) * ? + ? as delta_max,
    co.holding_days,
    AVG(co.contract_return_pct) as avg_return_pct,
    AVG(co.is_winner) as win_rate,
    COUNT(*) as sample_size
FROM recommended_contracts rc
JOIN contract_outcomes co ON co.recommended_contract_id = rc.id
WHERE rc.delta IS NOT NULL
  AND co.holding_days = ?
  AND co.contract_return_pct IS NOT NULL
GROUP BY CAST(rc.delta / ? AS INTEGER)
ORDER BY delta_min
```

6. **`get_performance_summary(lookback_days: int = 30) -> PerformanceSummary`**
```sql
-- Count total contracts in lookback window
-- Count contracts with outcomes
-- Compute overall win rate, avg returns
-- Find best direction and holding period
```
   - Multiple queries or one complex query with subqueries
   - Returns single `PerformanceSummary` model
   - `best_direction` / `best_holding_days`: determined by highest win rate

### Edge Cases
- No outcomes exist yet: all analytics queries return empty lists / None optional fields
- Single contract with single outcome: all queries should still work
- Division by zero in bucket calculations: composite_score=0, delta=0
- Very large lookback_days: should not cause performance issues with reasonable data
- Invalid indicator name in attribution query: return empty list
- Concurrent collect-outcomes requests: operation mutex prevents parallel runs
