# Task: Extract _debate_single() from _debate_async()

## Description
Refactor `src/options_arena/cli/commands.py` to extract the core debate logic from
`_debate_async()` into a reusable `_debate_single()` function. This is a pure refactor
with zero behavior change — the existing single-ticker debate path must call
`_debate_single()` and produce identical output.

The new function encapsulates: fetch quote + ticker info + chains → recommend contracts
→ run debate → return DebateResult. It receives pre-created services rather than creating
its own.

## Acceptance Criteria
- [ ] New `_debate_single()` async function exists in `commands.py`
- [ ] Signature: `async def _debate_single(ticker_score: TickerScore, settings: AppSettings, market_data: MarketDataService, options_data: OptionsDataService, fred: FredService, repo: Repository, *, fallback_only: bool = False) -> DebateResult`
- [ ] `_debate_async()` calls `_debate_single()` for the actual debate work
- [ ] `_debate_async()` still handles history mode, scan lookup, service lifecycle, and rendering
- [ ] Existing `debate AAPL` behavior is 100% preserved (no visible output change)
- [ ] All existing tests still pass
- [ ] `ruff check`, `ruff format`, `mypy --strict` all pass

## Technical Details
- Extract lines ~249-308 of current `_debate_async()` (from "Fetch live market data" through `run_debate()`) into `_debate_single()`
- `_debate_single()` returns `DebateResult` — rendering stays in `_debate_async()`
- The function does NOT create or close services — those are managed by the caller
- `_debate_single()` does NOT print status messages (`"Fetching live data..."`) — the caller handles that, or batch mode skips it
- The `fallback_only` config override logic (`DebateConfig` with near-zero timeout) moves into `_debate_single()` since both single and batch paths need it
