## Technical Details

### Integration test (`tests/unit/scan/test_filter_integration.py`):
Test the full pipeline with mocked services but real filter application:

1. Construct `ScanFilterSpec` with specific overrides (e.g., `min_score=40`, `sectors=["Technology"]`, `market_cap_tiers=["mega"]`)
2. Create `AppSettings` with the filter spec on `ScanConfig.filters`
3. Create `ScanPipeline` with mocked services
4. Run the pipeline
5. Verify:
   - Phase 1 applied market cap filter (fewer tickers than universe)
   - Phase 2 → scoring cutoff applied (fewer tickers than scored)
   - Phase 3 applied earnings check before chain fetch
   - Phase 4 persisted `filter_spec_json`

### CLI mapping test (`tests/unit/cli/test_scan_filter_args.py`):
Test that flat CLI args are correctly assembled into `ScanFilterSpec`:
- `--sector technology --sector healthcare` → `filters.universe.sectors == [GICSSector.TECHNOLOGY, GICSSector.HEALTHCARE]`
- `--top-n 30` → `filters.options.top_n == 30`
- `--min-score 40` → `filters.scoring.min_score == 40.0`
- `--min-confidence 0.5` → `filters.scoring.min_direction_confidence == 0.5`
- `--min-dte 30 --max-dte 90` → `filters.options.min_dte == 30, max_dte == 90`
- `--market-cap mega --market-cap large` → `filters.universe.market_cap_tiers == [MarketCapTier.MEGA, MarketCapTier.LARGE]`

### API mapping test (`tests/unit/api/test_scan_filter_request.py`):
Test that `ScanRequest` fields are correctly mapped to `ScanFilterSpec`:
- All existing fields preserve behavior
- New `min_direction_confidence` field maps correctly
- Default `ScanRequest()` produces default `ScanFilterSpec()`
