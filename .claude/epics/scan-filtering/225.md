# Task: Pre-scan universe narrowing controls

## Description

Add pre-scan universe narrowing via market cap tier filtering and earnings proximity exclusion. Extend `ScanConfig` with 4 new optional fields, wire them into the scan pipeline at appropriate phases (market cap in Phase 1, direction filter post-Phase 2, IV rank in Phase 3), update the API `ScanRequest` to accept them, add CLI options, and add ScanPage.vue controls for the web UI.

## Acceptance Criteria

- [ ] `ScanConfig` has 4 new fields: `market_cap_tiers: list[MarketCapTier]`, `exclude_near_earnings_days: int | None`, `direction_filter: SignalDirection | None`, `min_iv_rank: float | None`
- [ ] Pipeline Phase 1 applies market cap filter using existing `TickerInfo` data (tickers without TickerInfo pass through)
- [ ] Pipeline Phase 1 excludes tickers with earnings within N days (uses `next_earnings` from `TickerInfo`)
- [ ] Pipeline post-Phase 2 applies direction filter if set
- [ ] Pipeline Phase 3 applies min_iv_rank filter (after option chains fetched)
- [ ] `POST /api/scan` ScanRequest passes new fields to ScanConfig
- [ ] CLI `scan` command accepts `--market-cap`, `--exclude-earnings`, `--direction`, `--min-iv-rank` options
- [ ] ScanPage.vue has market cap MultiSelect and earnings proximity InputNumber
- [ ] All new fields have None/empty defaults (backward compatible)
- [ ] Omitting all filters runs the full universe (existing behavior preserved)

## Technical Details

### ScanConfig Extension (`models/config.py`)
```python
market_cap_tiers: list[MarketCapTier] = []  # empty = all tiers
exclude_near_earnings_days: int | None = None  # exclude within N days
direction_filter: SignalDirection | None = None  # filter post-Phase 2
min_iv_rank: float | None = None  # filter in Phase 3
```
- Validator for `market_cap_tiers`: deduplicate via `dict.fromkeys()` (same pattern as sectors)
- Validator for `min_iv_rank`: must be 0-100 if set

### Pipeline Filtering (`scan/pipeline.py`)

**Phase 1 — Market Cap Filter:**
- After universe fetch, if `config.market_cap_tiers` is non-empty:
  - Use existing `TickerInfo` data (fetched for enrichment) to check `market_cap_tier`
  - Tickers without TickerInfo pass through (no false exclusions per PRD decision)
  - Log count of filtered tickers

**Phase 1 — Earnings Proximity Filter:**
- If `config.exclude_near_earnings_days` is set:
  - Use `next_earnings` date from TickerInfo
  - Exclude tickers where `(next_earnings - today).days <= exclude_near_earnings_days`
  - Tickers without `next_earnings` pass through

**Post-Phase 2 — Direction Filter:**
- If `config.direction_filter` is set:
  - Filter scored tickers to matching direction only

**Phase 3 — IV Rank Filter:**
- If `config.min_iv_rank` is set:
  - After option chains are fetched and IV analytics computed
  - Filter tickers where `iv_rank < min_iv_rank`

### API Schema Extension (`api/schemas.py`)
- `ScanRequest.market_cap_tiers: list[MarketCapTier] = []`
- `ScanRequest.exclude_near_earnings_days: int | None = None`
- `ScanRequest.direction_filter: SignalDirection | None = None`
- `ScanRequest.min_iv_rank: float | None = None`
- Map these to `ScanConfig` fields when starting pipeline

### CLI Extension (`cli/scan.py`)
- `--market-cap`: Multiple choice from MarketCapTier values
- `--exclude-earnings`: Integer (days)
- `--direction`: Choice from SignalDirection values
- `--min-iv-rank`: Float (0-100)

### ScanPage.vue Extension
- PrimeVue `MultiSelect` for market cap tiers (Mega/Large/Mid/Small/Micro)
- PrimeVue `InputNumber` for exclude_near_earnings_days
- Place in scan setup form alongside existing preset/sector controls
- Pass values in POST /api/scan request body

### Edge Cases
- Empty `market_cap_tiers` list → no market cap filtering (all pass)
- `exclude_near_earnings_days=0` → exclude tickers with earnings today
- All tickers filtered out → pipeline handles empty universe gracefully
- `min_iv_rank=0` → effectively no filter (all pass)
- `min_iv_rank=100` → almost nothing passes
- Direction filter on ticker with neutral direction
