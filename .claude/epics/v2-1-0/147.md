# Task: Score History Backend

## Description

Implement the data and API layer for score history: a `HistoryPoint` model, repository methods
for fetching score history and trending tickers, and a new `routes/ticker.py` with history
endpoint. This provides the backend for the Score History Frontend (Task 6).

## Acceptance Criteria

- [ ] `HistoryPoint` Pydantic model: `scan_id`, `scan_date: datetime`, `composite_score: float`, `direction: str`, `preset: str`
- [ ] `TrendingTicker` Pydantic model: `ticker`, `direction`, `consecutive_scans: int`, `latest_score: float`, `score_change: float`
- [ ] `get_score_history(ticker, limit=20) -> list[HistoryPoint]` repository method
  - Joins `ticker_scores` + `scan_runs` on `scan_id`
  - Returns ordered by scan date descending
  - Limit defaults to 20 (last 20 scans)
- [ ] `get_trending_tickers(direction: str, min_scans: int = 3) -> list[TrendingTicker]` repository method
  - Finds tickers with consistent direction over `min_scans` consecutive scans
  - Returns sorted by `consecutive_scans` descending
- [ ] New `routes/ticker.py` with `GET /api/ticker/{ticker}/history` returning `list[HistoryPoint]`
- [ ] `GET /api/ticker/trending?direction=bullish&min_scans=3` returning `list[TrendingTicker]`
- [ ] ~20 Python tests: model validation, repository queries, API endpoints

### Implementation Approach

1. **HistoryPoint model**: `frozen=True`, UTC validator on `scan_date`, `composite_score`
   validated with `math.isfinite()`. Direction uses existing `DirectionSignal` enum.

2. **Repository `get_score_history`**:
   ```sql
   SELECT ts.scan_id, sr.started_at, ts.composite_score, ts.direction, sr.preset
   FROM ticker_scores ts
   JOIN scan_runs sr ON ts.scan_id = sr.id
   WHERE ts.ticker = ?
   ORDER BY sr.started_at DESC
   LIMIT ?
   ```
   Returns `list[HistoryPoint]` constructed from rows.

3. **Repository `get_trending_tickers`**: Query last N scans per ticker, check for consecutive
   same-direction entries. This is more complex — use a window function or application-level
   grouping. Start with application-level grouping for clarity:
   - Get all tickers from latest scan
   - For each, check last `min_scans` entries have same direction
   - Filter and sort by streak length

4. **API routes**: Thin handlers, `Depends(get_repo)` for repository injection.

### Key Considerations
- Score history query should be fast (< 100ms for 20-scan lookback) — index on `ticker_scores.ticker`
  already exists from initial schema
- Trending tickers query may need optimization if ticker count is high — limit to tickers in latest scan
- Read `data/CLAUDE.md` and `api/CLAUDE.md` before implementation
