# PRD: v2.1.0 — Close the Loop

## Executive Summary

Options Arena v2.0.0 delivers a complete **screen → analyze** workflow (scan pipeline + AI debate). But the loop stops there — users have no way to **track** tickers over time, **compare** scan results, or get **historical context** on their analysis. v2.1.0 closes this gap with five features that enable the full **screen → analyze → decide → track** cycle.

The release theme is **"Close the Loop"**: give returning users immediate productivity ("what changed?"), let them bookmark and monitor tickers across sessions, surface historical score trends, reduce friction on the most common workflow (ad-hoc debate), and add a safety rail against earnings-related IV crush.

All five features share existing data infrastructure (`scan_runs`, `ticker_scores`, `watchlists`/`watchlist_tickers` tables), require zero schema migrations, and compound with each other.

## Problem Statement

After scanning and debating, users hit a dead end:

1. **No tracking** — The `watchlists` and `watchlist_tickers` tables exist but have zero repository methods, zero API endpoints, and zero UI. Users cannot bookmark tickers or revisit past analysis.
2. **No historical context** — Score data accumulates across scan runs but is never surfaced. Users cannot answer "Is AAPL trending up or down over the last 5 scans?"
3. **No scan diffing** — Returning users must mentally compare two scan result pages to see what changed.
4. **Workflow friction** — Starting a debate requires navigating to scan results first. Power users want to type a ticker and go.
5. **No earnings awareness** — Debating a ticker reporting earnings tomorrow is dangerous (IV crush), and there's no warning anywhere in the system.

These gaps make Options Arena a single-session tool rather than a daily-use platform.

## User Stories

### US-1: Watchlist Management
**As a** returning user, **I want to** save tickers to named watchlists **so that** I can quickly revisit tickers I'm monitoring without re-scanning.

**Acceptance Criteria:**
- Can create, rename, and delete watchlists via Web UI and CLI
- Can add/remove tickers to/from a watchlist from scan results table and TickerDrawer
- Watchlist page shows each ticker's latest score, direction, and last debate date
- Watchlists persist across sessions (SQLite)
- CLI: `options-arena watchlist list`, `options-arena watchlist add AAPL --name "Tech Picks"`

### US-2: Scan Delta View
**As a** daily user, **I want to** compare two scan runs side-by-side **so that** I can immediately see what changed since my last scan.

**Acceptance Criteria:**
- "Compare with..." dropdown on ScanResultsPage populated from scan history
- Delta chips: green up-arrow for improved scores, red down-arrow for drops
- "NEW" badge for tickers appearing in current scan but not the baseline
- Shows added tickers, removed tickers, and top movers (sorted by absolute score change)

### US-3: Quick Debate from Dashboard
**As a** power user, **I want to** type a ticker on the dashboard and start a debate immediately **so that** I don't have to navigate through scan results first.

**Acceptance Criteria:**
- InputText + "Debate" button on the DashboardPage quick-actions bar
- Submits `POST /api/debate` with ticker (no scan_id required — API already supports this)
- Shows existing DebateProgressModal via WebSocket
- Auto-navigates to `/debate/{id}` on completion

### US-4: Score History Timeline
**As an** analyst, **I want to** see how a ticker's composite score has trended over past scans **so that** I can assess momentum and make more informed decisions.

**Acceptance Criteria:**
- SVG line chart: scan dates on X-axis, composite score on Y-axis
- Direction encoded as dot color (green=bullish, red=bearish, yellow=neutral)
- Rendered in TickerDrawer and on a dedicated `/ticker/{ticker}` page
- Dashboard integration: "Trending Up" / "Trending Down" sections (3+ consecutive scans)
- Sparkline previews in scan results DataTable for inline context

### US-5: Earnings Calendar Overlay
**As a** risk-conscious trader, **I want to** see upcoming earnings dates for scanned tickers **so that** I avoid debating tickers with imminent IV crush risk.

**Acceptance Criteria:**
- New `fetch_earnings_date(ticker) -> date | None` in `MarketDataService` (yfinance `Ticker.calendar`, 24h cache TTL)
- Scan results table: "Earnings" column showing DTE to next earnings ("3d" in red if < 7 days, "45d" in gray otherwise)
- TickerDrawer: earnings warning banner if < 7 days
- Debate agent prompts: inject "WARNING: Earnings in {N} days. IV crush likely." when < 7 days
- `MarketContext.next_earnings_date: date | None` field populated during Phase 3

## Requirements

### Functional Requirements

#### FR-1: Watchlist CRUD (Tier 1, Effort: M)
- **Data layer**: 6 new repository methods in `data/repository.py`:
  - `create_watchlist(name: str) -> Watchlist`
  - `delete_watchlist(watchlist_id: int) -> None`
  - `add_ticker_to_watchlist(watchlist_id: int, ticker: str) -> None`
  - `remove_ticker_from_watchlist(watchlist_id: int, ticker: str) -> None`
  - `get_watchlists() -> list[Watchlist]`
  - `get_tickers_for_watchlist(watchlist_id: int) -> list[WatchlistTicker]`
- **API**: New `routes/watchlist.py` with REST endpoints:
  - `POST /api/watchlist` — create watchlist
  - `DELETE /api/watchlist/{id}` — delete watchlist
  - `GET /api/watchlist` — list all watchlists
  - `POST /api/watchlist/{id}/tickers` — add ticker
  - `DELETE /api/watchlist/{id}/tickers/{ticker}` — remove ticker
  - `GET /api/watchlist/{id}` — get watchlist with tickers
- **Web**: New `WatchlistPage.vue` at `/watchlist` route. "Add to Watchlist" button in `TickerDrawer.vue` and scan results context menu. PrimeVue DataTable with ticker, score snapshot, direction badge, last debate date.
- **CLI**: `options-arena watchlist` subcommand group (`list`, `add`, `remove`, `create`, `delete`)
- **Schema migration**: None — `watchlists` and `watchlist_tickers` tables exist since `001_initial.sql`

#### FR-2: Scan Delta View (Tier 1, Effort: S-M)
- **Models**: New `TickerDelta` and `ScanDiff` Pydantic models
- **API**: `GET /api/scan/{id}/diff?base_id={prev_id}` returns `ScanDiff` with `added`, `removed`, and `movers` (sorted by abs score change)
- **Repository**: Leverage existing `get_scores_for_scan()`. Diff computed in route handler via two queries + set operations + score comparison.
- **Web**: "Compare with..." dropdown on `ScanResultsPage.vue` (populated from `scanStore.scans`). Delta chips (green up-arrow, red down-arrow), "NEW" badge for added tickers.

#### FR-3: Quick Debate from Dashboard (Tier 1, Effort: S)
- **Web**: `InputText` + "Debate" `Button` in `DashboardPage.vue` quick-actions bar
- **Integration**: Calls existing `POST /api/debate` (already supports `scan_id: int | None`)
- **WebSocket**: Connects to `/ws/debate/{id}`, shows `DebateProgressModal`
- **Navigation**: Auto-navigate to `/debate/{id}` on completion

#### FR-4: Score History Timeline (Tier 2, Effort: M)
- **Repository**: New `get_score_history(ticker: str, limit: int = 20) -> list[HistoryPoint]` joining `ticker_scores` + `scan_runs`
- **API**: `GET /api/ticker/{ticker}/history` returns `list[HistoryPoint]` with `(scan_date, composite_score, direction)` tuples
- **Web**: New `ScoreHistoryChart.vue` component (SVG line chart). Rendered in `TickerDrawer` and new `/ticker/{ticker}` page.
- **Dashboard**: "Trending Up" / "Trending Down" sections showing tickers with consistent score movement over 3+ scans
- **Sparklines**: Inline SVG sparklines in scan results DataTable for each ticker

#### FR-5: Earnings Calendar Overlay (Tier 2, Effort: M)
- **Service**: `MarketDataService.fetch_earnings_date(ticker) -> date | None` using yfinance `Ticker.calendar`, wrapped in `asyncio.to_thread`, 24h cache TTL
- **Model**: New optional field `MarketContext.next_earnings_date: date | None`
- **Scan pipeline**: Fetch earnings date in Phase 3 alongside ticker info
- **Web**: "Earnings" column in scan results table (DTE display, red if < 7d). Warning banner in TickerDrawer.
- **Debate integration**: Inject earnings proximity warning into agent prompts when < 7 days

### Non-Functional Requirements

- **No schema migrations**: All five features use existing tables or ephemeral data. Zero DDL changes.
- **Performance**: Score history queries must complete in < 100ms for 20-scan lookback. Earnings fetches cached with 24h TTL to avoid yfinance rate limits.
- **Backward compatibility**: All new API endpoints are additive. Existing CLI commands and API contracts unchanged.
- **Test coverage**: Each feature requires unit tests for repository methods, API endpoint integration tests, and Playwright E2E tests for new UI components.
- **Accessibility**: New UI components follow existing PrimeVue patterns (keyboard navigation, ARIA labels).
- **Mobile responsiveness**: New pages (Watchlist, Compare) must render acceptably on tablet-width viewports (768px+).

## Constraints & Assumptions

### Constraints
- **No new external services**: All features use existing data sources (SQLite, yfinance). No new API keys or paid services.
- **yfinance earnings data**: Coverage varies — some tickers may not have earnings dates in yfinance's `calendar` property. Graceful `None` handling required.
- **SVG charts only**: No charting library dependency. Score history and sparklines use hand-crafted SVG to avoid adding chart.js/d3 to the bundle.
- **Single-process architecture**: No background workers or scheduled scans in v2.1.0. Alerts (#2.4) deferred to v2.2.0 precisely because they need a scheduler.

### Assumptions
- Users run scans at least weekly, so score history accumulates naturally.
- The existing `watchlists`/`watchlist_tickers` schema from `001_initial.sql` is adequate (no schema changes needed).
- yfinance `Ticker.calendar` returns earnings dates reliably for S&P 500 tickers.

|---------|-------------|

| **Options Chain Explorer** (#2.3) | Large frontend component, decoupled from "track" theme |
| **Alert System** (#2.4) | Requires schema migration + WebSocket channel + scheduler; better as own release |
| **Multi-Provider LLM** (#3.2) | Large refactor, independent of user workflow improvements |
| **Incremental Scan** (#3.1) | Performance optimization; ship after workflow gaps are closed |
| **Event Bus** (#3.3) | Architecture upgrade, not user-facing; defer until callback pain is acute |
| **Frontend unit tests** (Vitest) | Desirable but not blocking; E2E coverage is sufficient for v2.1.0 |

## Dependencies

### Internal
- **Existing schema**: `watchlists`, `watchlist_tickers` tables from `001_initial.sql`
- **Existing data**: `scan_runs`, `ticker_scores` tables for history and delta queries
- **Existing API**: `POST /api/debate` already supports ticker-only debates (no scan_id)
- **Existing services**: `MarketDataService`, `OptionsDataService`, `ServiceCache` for earnings data

### External
- **yfinance**: `Ticker.calendar` for earnings dates (free, already a dependency)
- **No new packages**: All features implementable with existing Python and JS dependencies

### Feature Interdependencies
```
FR-3 (Quick Debate)     → standalone, no deps
FR-1 (Watchlist CRUD)   → standalone, no deps
FR-2 (Scan Delta)       → standalone, no deps
FR-4 (Score History)    → compounds with FR-1 (sparklines on watchlist page)
FR-5 (Earnings)         → standalone, enriches debate prompts
FR-4 (Dashboard trends) → reuses FR-2 (scan delta) logic for "top movers"
```

### Recommended Implementation Order
1. **FR-3** Quick Debate (S) — smallest, standalone, immediate UX win
2. **FR-1** Watchlist CRUD (M) — foundation for tracking loop
3. **FR-2** Scan Delta View (S-M) — shared logic needed by FR-4 dashboard
4. **FR-4** Score History Timeline (M) — builds on FR-1 + FR-2
5. **FR-5** Earnings Calendar Overlay (M) — independent, can parallelize with FR-4

Total estimated scope: ~5 weeks, parallelizable across frontend/backend tracks.
