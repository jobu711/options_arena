# Options Arena V2 — Implementation Plan

## Context

Options Arena V1 (MVP 1.0.0) is complete: 8 phases, 1,086 tests, full scan pipeline
producing 8 recommendations per run via CLI. The agents module is **scaffolded but empty**
(only CLAUDE.md files, zero Python code). The `ai_theses` table exists in the migration
but is unused. All analysis models (`MarketContext`, `AgentResponse`, `TradeThesis`) are
implemented and tested.

**V2 Goal:** Add the two features that define Options Arena's unique value — AI-powered
debate on individual tickers and historical performance tracking — delivered through a
Python-native web dashboard (FastAPI + HTMX + Jinja2).

## Design Decisions (from user interview)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Audience | Personal power tool | Dense UI, no onboarding, keyboard shortcuts |
| Interface | Web dashboard | FastAPI + HTMX + Jinja2 (Python-native) |
| Core features | AI debate + Historical tracking | The moat — no competitor does local AI debate |
| Debate flow | On-demand, single-pass | Bull -> Bear -> Risk. ~90s on CPU. Click to trigger. |
| Data freshness | Batch | Run scan, view results, no live updates |
| Ollama | Local CPU (~60s/debate) | Cache debates aggressively, excellent loading states |
| Historical tracking | Full (P&L + score predictiveness + trends) | Manual reconciliation trigger |
| Visualization | Deferred to V2.1 | Charts, P&L diagrams are nice-to-have after core features ship |
| Strategy builder | Deferred to V3 | Multi-leg spreads, iron condors — too complex for V2 |

## Implementation Phases

### Phase 9: AI Debate System (~8 issues)

**Goal:** Implement Bull/Bear/Risk single-pass debate using PydanticAI + Ollama.

**Why first:** The debate system is backend-only. Build and test it before adding UI.
Can verify via CLI `options-arena debate AAPL` before web UI exists.

**Files to create:**
- `src/options_arena/agents/__init__.py` — re-exports
- `src/options_arena/agents/model_config.py` — `build_ollama_model()`, host resolution
- `src/options_arena/agents/bull.py` — PydanticAI `Agent[DebateDeps, AgentResponse]`
- `src/options_arena/agents/bear.py` — PydanticAI `Agent[DebateDeps, AgentResponse]`
- `src/options_arena/agents/risk.py` — PydanticAI `Agent[DebateDeps, TradeThesis]`
- `src/options_arena/agents/_parsing.py` — shared models, constants, think-tag validator
- `src/options_arena/agents/orchestrator.py` — single-pass debate coordinator
- `src/options_arena/agents/prompts/templates/` — `.txt` prompt files (bull, bear, risk)

**Files to modify:**
- `src/options_arena/data/repository.py` — add `save_debate()`, `get_debates_for_ticker()`
- `src/options_arena/cli/commands.py` — add `debate` command
- `src/options_arena/models/config.py` — add `DebateConfig` nested model if needed

**Key patterns (from agents/CLAUDE.md):**
- Module-level `Agent[Deps, OutputType]` instances (not classes)
- `@dataclass` deps: `MarketContext` + prior agent responses
- `output_type=AgentResponse` (bull/bear), `output_type=TradeThesis` (risk)
- `retries=2`, `model_settings=ModelSettings(extra_body={"num_ctx": 8192})`
- `@agent.output_validator` rejects `<think>` tag remnants
- Data-driven fallback when Ollama unreachable (synthesize verdict from scores)
- `RunUsage` accumulation for token tracking

**Debate flow:**
```
1. Build MarketContext from TickerScore + live quote + ticker info
2. Bull agent: argue bullish case → AgentResponse
3. Bear agent: receive bull's argument, argue bearish → AgentResponse
4. Risk agent: receive both, synthesize → TradeThesis
5. Persist to ai_theses table
6. Return DebateResult (context + responses + verdict + usage metadata)
```

**Testing:** ~80-100 tests using PydanticAI `TestModel` (no real Ollama needed).

### Phase 10: Web Foundation (~6 issues)

**Goal:** FastAPI + HTMX skeleton with scan dashboard, health page, and base layout.

**Why second:** Gives us the web shell to build debate UI and history dashboard on.

**New dependencies:** `fastapi`, `uvicorn`, `jinja2`, `python-multipart`
(HTMX is a single JS file served from CDN or vendored — no npm)

**Files to create:**
```
src/options_arena/web/
    __init__.py
    app.py              — FastAPI app factory, lifespan, static files
    routes/
        __init__.py
        dashboard.py    — GET / (latest scan results table)
        scan.py         — POST /scan (trigger scan), GET /scan/{id}
        health.py       — GET /health (service status)
        universe.py     — GET /universe (stats, sector breakdown)
    templates/
        base.html       — Layout: nav, content area, footer w/ disclaimer
        dashboard.html  — Scan results table (HTMX-powered)
        scan_detail.html — Single scan run detail
        health.html     — Health check status
        universe.html   — Universe stats
        _partials/
            scan_table.html   — Reusable scan results partial
            progress.html     — Scan progress indicator (SSE)
    static/
        htmx.min.js     — Vendored HTMX
        style.css        — Minimal custom CSS (Tailwind CDN or Pico CSS)
    dependencies.py      — FastAPI dependency injection (services, settings)
    CLAUDE.md            — Module conventions
```

**Files to modify:**
- `pyproject.toml` — add FastAPI, uvicorn, jinja2 deps
- `src/options_arena/cli/commands.py` — add `serve` command (`options-arena serve`)
- `src/options_arena/cli/app.py` — register serve command

**Key design:**
- `lifespan` context manager creates/closes services (same DI as CLI)
- Templates extend `base.html` with `{% block content %}`
- HTMX `hx-get`, `hx-post`, `hx-swap` for interactivity without JS
- SSE endpoint for scan progress (`EventSourceResponse`)
- All routes return `TemplateResponse` (server-rendered, no SPA)
- Scan results table mirrors CLI output but with clickable ticker links

**Entry point:** `options-arena serve --port 8000` starts uvicorn

### Phase 11: Debate UI + Ticker Detail (~5 issues)

**Goal:** Ticker deep-dive page with indicator breakdown and on-demand AI debate.

**Files to create:**
```
src/options_arena/web/
    routes/
        ticker.py       — GET /ticker/{symbol} (detail page)
        debate.py       — POST /debate/{symbol} (trigger debate, SSE stream)
    templates/
        ticker_detail.html — Score breakdown, contracts, debate trigger
        debate.html        — Debate results display
        _partials/
            indicator_grid.html  — 18 indicators grouped by category
            contract_table.html  — Recommended contracts with Greeks
            debate_stream.html   — SSE-updated debate progress
            debate_result.html   — Final verdict display
```

**Debate UX flow (critical — CPU is slow):**
```
1. User clicks ticker in scan table → /ticker/AAPL
2. Page shows: composite score, direction, 18 indicators, recommended contracts
3. User clicks "Run Debate" button
4. Button replaced with progress indicator via HTMX
5. SSE endpoint streams: "Bull agent thinking..." → bull response partial
   → "Bear agent thinking..." → bear response partial
   → "Risk agent synthesizing..." → verdict partial
6. Final state: full debate result with Bull/Bear/Risk panels + verdict card
7. Result cached in ai_theses — subsequent visits show cached debate
```

**SSE streaming pattern:**
```python
@router.post("/debate/{ticker}")
async def run_debate(ticker: str) -> EventSourceResponse:
    async def event_generator():
        yield {"event": "phase", "data": "bull"}
        bull_result = await orchestrator.run_bull(context)
        yield {"event": "result", "data": bull_result.model_dump_json()}
        # ... bear, risk phases
    return EventSourceResponse(event_generator())
```

**HTMX receives SSE events and swaps partials into the page.**

### Phase 12: Historical Tracking (~6 issues)

**Goal:** Track recommendation outcomes, calculate performance metrics, display trends.

**New migration:** `data/migrations/003_historical_tracking.sql`
```sql
-- Recommendation outcomes (entry + exit tracking)
CREATE TABLE recommendation_outcomes (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    scan_run_id INTEGER NOT NULL REFERENCES scan_runs(id),
    ticker TEXT NOT NULL,
    option_type TEXT NOT NULL,
    strike REAL NOT NULL,
    expiration TEXT NOT NULL,
    entry_mid REAL NOT NULL,
    entry_date TEXT NOT NULL,
    exit_mid REAL,
    exit_date TEXT,
    status TEXT NOT NULL DEFAULT 'open',  -- open, profit, loss, expired
    pnl REAL,
    pnl_pct REAL,
    underlying_entry_price REAL NOT NULL,
    underlying_exit_price REAL,
    composite_score REAL NOT NULL,
    direction TEXT NOT NULL
);

-- Aggregated performance metrics
CREATE TABLE performance_metrics (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    calculated_at TEXT NOT NULL,
    period_start TEXT NOT NULL,
    period_end TEXT NOT NULL,
    total_recs INTEGER NOT NULL,
    win_rate REAL,
    avg_pnl_pct REAL,
    score_correlation REAL  -- composite_score vs actual P&L
);
```

**Files to create:**
```
src/options_arena/services/reconciliation.py  — fetch current prices for open recs
src/options_arena/web/routes/history.py       — GET /history, GET /history/{scan_id}
src/options_arena/web/templates/
    history.html          — Performance dashboard
    history_detail.html   — Single scan outcome detail
    _partials/
        outcome_table.html     — Recommendations with P&L
        performance_card.html  — Win rate, avg P&L, trend
        score_chart.html       — Score vs outcome scatter (future)
```

**Files to modify:**
- `src/options_arena/data/repository.py` — outcome CRUD methods
- `src/options_arena/scan/pipeline.py` — save recommendations to `recommendation_outcomes` at entry
- `src/options_arena/cli/commands.py` — add `reconcile` command

**Reconciliation flow:**
```
1. User runs: options-arena reconcile (CLI) or clicks button (web)
2. Fetch all 'open' outcomes from DB
3. For each: fetch current option price via yfinance
4. Update: current P&L, status (expired if past expiration)
5. Recalculate performance_metrics
6. Display updated dashboard
```

**Score predictiveness tracking:**
- For each scored ticker, record underlying price at scan time
- At reconciliation, check price 7/14/30 days later
- Did BULLISH tickers go up? Did BEARISH go down?
- Correlation metric: composite_score vs magnitude of correct directional move

## What's NOT in V2

- Multi-round debate (single-pass only)
- P&L payoff diagrams / Greeks surfaces (V2.1 — add charting lib later)
- Strategy builder / multi-leg spreads (V3)
- Real-time WebSocket streaming (batch is sufficient)
- Mobile optimization (desktop-first personal tool)
- Additional LLM providers beyond Ollama (V2.1)
- Automated scheduled reconciliation (manual trigger only)
